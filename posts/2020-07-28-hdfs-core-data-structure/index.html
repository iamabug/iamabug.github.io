NameNode 文件系统相关数据结构初探

 2020.7.28  2021.2.5  格物  1679  4 mins  

如前文所述，文件系统对象 FSNamesystem 是NameNode最核心的数据结构，所以，本文是我对它的学习过程和总结。

我的学习方法是，先过一遍相关的源码，心中有个大概的轮廓，然后去搜索相关的关键词对已有的认知进行扩充和完善，最后完成进行总结。

/**
 * Instantiates an FSNamesystem loaded from the image and edits
 * directories specified in the passed Configuration.
 *
 * @param conf the Configuration which specifies the storage directories
 *             from which to load
 * @return an FSNamesystem which contains the loaded namespace
 * @throws IOException if loading fails
 */
static FSNamesystem loadFromDisk(Configuration conf) throws IOException {
  // 检查配置
  checkConfiguration(conf);
  // 创建FSImage对象，我先把它简单地看做是相关文件目录在内存中的抽象
  FSImage fsImage = new FSImage(conf,
      FSNamesystem.getNamespaceDirs(conf),
      FSNamesystem.getNamespaceEditsDirs(conf));
  // 创建FSNamesystem对象
  FSNamesystem namesystem = new FSNamesystem(conf, fsImage, false);
  
  // 通过FSImage初始化FSNamesystem对象
  namesystem.loadFSImage(startOpt);
  
  // 这里是前文提到过的NameNode的metric对象，更新其中的FsImage加载时间
  NameNodeMetrics nnMetrics = NameNode.getNameNodeMetrics();
  if (nnMetrics != null) {
    nnMetrics.setFsImageLoadTime((int) timeTakenToLoadFSImage);
  }
  return namesystem;
}

如函数注释所说，这个函数读取FsImage以及edit log来初始化 FSNamesystem 对象，下面再来看下 FSNamesystem 的构造函数。

/**
 * Create an FSNamesystem associated with the specified image.
 * 
 * Note that this does not load any data off of disk -- if you would
 * like that behavior, use {@link #loadFromDisk(Configuration)}
 *
 * @param conf configuration
 * @param fsImage The FSImage to associate with
 * @param ignoreRetryCache Whether or not should ignore the retry cache setup
 *                         step. For Secondary NN this should be set to true.
 * @throws IOException on bad configuration
 */
FSNamesystem(Configuration conf, FSImage fsImage, boolean ignoreRetryCache)
    throws IOException {
  // 初始化锁相关数据结构
  boolean fair = conf.getBoolean("dfs.namenode.fslock.fair", true);
  fsLock = new FSNamesystemLock(fair);
  cond = fsLock.writeLock().newCondition();
  cpLock = new ReentrantLock();
  
  // FSimage引用传递
  this.fsImage = fsImage;
  
  // 初始化三个重要的数据结构
  this.blockManager = new BlockManager(this, conf);
  this.datanodeStatistics = blockManager.getDatanodeManager().getDatanodeStatistics();
  this.blockIdManager = new BlockIdManager(blockManager);
  
  // 初始化文件系统的基本信息
  this.fsOwner = UserGroupInformation.getCurrentUser();
  this.supergroup = conf.get(DFS_PERMISSIONS_SUPERUSERGROUP_KEY, 
                             DFS_PERMISSIONS_SUPERUSERGROUP_DEFAULT);
  this.isPermissionEnabled = conf.getBoolean(DFS_PERMISSIONS_ENABLED_KEY,
                                                 DFS_PERMISSIONS_ENABLED_DEFAULT);
  nameserviceId = DFSUtil.getNamenodeNameServiceId(conf);
  this.haEnabled = HAUtil.isHAEnabled(conf, nameserviceId);
  this.dir = new FSDirectory(this, conf);
  
  // 配置及其它数据结构的初始化
  // ...
}

注释比较清晰，构造函数不会从FSImage中加载数据，需要调用 loadFromDisk，它也确实调用了。

FSNamesystem组成

通过上面的代码已经可以看到 FSNamesystem内部的结构，下面分别看各个组件。

FSDirectory

类注释：

1
2
3
4
5
6
7

/**
 * Both FSDirectory and FSNamesystem manage the state of the namespace.
 * FSDirectory is a pure in-memory data structure, all of whose operations
 * happen entirely in memory. In contrast, FSNamesystem persists the operations
 * to the disk.
 * @see org.apache.hadoop.hdfs.server.namenode.FSNamesystem
 **/

如注释所说，FSDirectory 和 FSNamesystem 都用来管理文件系统的状态，只不过 FSDirectory 是纯内存的数据结构，它的所有操作都在内存中完成，而FSNamesystem 会将操作持久化到磁盘中。

FSDirectory 类的重要类成员有：

1
2

INodeDirectory rootDir;
private final INodeMap inodeMap;

其中，文件系统是一棵树，每个节点是一个 INode 对象，每个对象有一个唯一ID，INode 有表示目录和文件的子类，rootDir 是这棵树的根节点，同时，因为在树中查找文件效率较低，需要一个字典来弥补树的不足，这个字典就是 INodeMap 对象，从ID映射到 INode 对象。
